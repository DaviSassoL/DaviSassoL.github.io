Pipeline de Dados & IA para Treinos Personalizados

Projeto de arquitetura de dados que integra múltiplas fontes de informação de treinos físicos — API Hevy, formulários de usuários e datasets externos — em um Data Lake em nuvem. A solução aplica os conceitos pedidos no Hands-on: arquitetura Lambda, camadas Bronze–Silver–Gold, modelagem analítica, governança, CI/CD e Infraestrutura como Código. Na camada de consumo, um app Streamlit com IA generativa posiciona o usuário no centro da jornada, mostrando a evolução de treinos de forma clara e orientada a dados.

Cenário do cliente

Uma pessoa que treina regularmente registra as sessões em diferentes sistemas, planilhas e anotações. Os dados existem, mas não se transformam em um ativo estratégico. Falta uma visão integrada que responda perguntas simples, como: “estou evoluindo na carga do supino?”, “meu volume semanal está consistente?” ou “como estou em relação à média de outros praticantes?”.

Do ponto de vista do instrutor, a personalização dos treinos depende de memória, experiência e consultas manuais a registros espalhados. O objetivo deste projeto é propor uma esteira de dados que unifique essas informações, gere métricas confiáveis e permita uma experiência analítica apoiada por IA — da fonte ao insight.

Contexto do Projeto

Setor, dor e oportunidade

Setor – Saúde, Fitness & Bem-estar
  • Mercado em crescimento, com apps de treino, wearables e plataformas digitais.
  • Usuários registram treinos, sono, nutrição e métricas corporais em diversos sistemas, gerando grandes volumes de dados pouco explorados.
  • Empresas e instrutores procuram maneiras de transformar esses dados em planos mais personalizados, aumento de aderência e redução de abandono.
Problema de Negócio
  • Ausência de uma visão única do histórico de treinos de cada usuário.
  • Dificuldade para acompanhar evolução de cargas, volume e frequência de treino de forma visual e confiável.
  • Comparações com médias globais ou grupos de praticantes quase inexistentes.
  • Decisões de ajuste de treino baseadas mais em percepção subjetiva do que em dados.

Pergunta orientadora do projeto: como uma arquitetura de dados em nuvem, bem governada, pode tornar o histórico de treinos um insumo estratégico para decisões de usuários e instrutores?

Slide 3 · Objetivos Próximo: visão geral da solução
Visão Geral

Da coleta ao insight para o usuário

1 · Fontes de Dados
  • API Hevy – workouts, séries e exercícios em JSON, representando o comportamento real do praticante ao longo do tempo.
  • Formulário Streamlit – captura de perfil, objetivos, restrições e contexto do usuário (por exemplo, foco em hipertrofia, saúde geral ou performance).
  • Datasets externos – Kaggle e dados sintéticos para complementar catálogo de exercícios e permitir testes mesmo sem conexão com a API.
2 · Pipeline & Orquestração
  • Apache Airflow coordena a ingestão, as transformações e a carga nas camadas do Data Lake, garantindo dependências e reexecução controlada.
  • Scripts em Python e Pandas fazem a limpeza, padronização, cálculo de métricas e enriquecimento dos dados, evoluindo de Bronze para Silver e, depois, Gold.
  • O pipeline é pensado como um grafo de tarefas, não como scripts isolados, alinhado à proposta de arquitetura do professor.
3 · Consumo & Experiência
  • Um app Streamlit lê a camada Gold para produzir dashboards, relatórios e visualizações simples para o usuário final.
  • Um chat de IA, baseado em LLaMA 3.1 via Groq, usa o perfil e o histórico para responder perguntas e sugerir ajustes nos treinos.
  • A estrutura da Gold permite conectar ferramentas de BI como Power BI, Redshift ou Looker, caso a solução seja levada para um ambiente corporativo.

Em termos de storytelling: começamos em dados brutos dispersos, passamos por uma esteira automatizada de tratamento e terminamos em uma experiência em que o usuário enxerga sua jornada de treino com clareza, suportada por IA.

Slide 4 · Visão Geral da Arquitetura Próximo: Arquitetura Lambda detalhada
Arquitetura Lambda

Camadas propostas no Hands-on, aplicadas ao caso

Source & Data Acquisition
  • Source Systems: API Hevy, formulário Streamlit, datasets Kaggle e dados sintéticos gerados por script.
  • Data Acquisition Layer: scripts Python que consomem as APIs, leem arquivos externos e organizam os dados para ingestão.
  • Essa etapa trata autenticação, paginação, limites de API e normalização básica de esquemas.
Batch / Processing Layer
  • Airflow implementa a camada de processamento batch, conectando as tasks create_lake_structure → ingest_* → silver_* → gold_metrics.
  • Transformações com Pandas realizam limpeza, enriquecimento e cálculos de métricas de treino, seguindo o modelo de camadas Bronze/Silver/Gold sugerido pelo professor.
  • O objetivo é chegar a dados prontos para análise OLAP e para um possível Wide Table voltado a machine learning.
Storage & Serving Layer
  • Data Storage Layer: AWS S3 como Data Lake, com estrutura em pastas por camada e domínio de negócio.
  • Serving Layer: Streamlit + possíveis ferramentas de BI consumindo a Gold; APIs adicionais podem ser construídas sobre o mesmo modelo de dados.
  • A arquitetura respeita o desenho proposto no material: das fontes, passando por processamento batch, até a apresentação dos dados ao usuário final.
Slide 5 · Arquitetura Lambda Próximo: Bronze, Silver e Gold em detalhes
Data Lake em AWS S3

Camadas Bronze, Silver e Gold

Bronze – Raw
  • Dados brutos exatamente como chegam das fontes, preservando a fidelidade ao sistema de origem.
  • Pastas organizadas por domínio, por exemplo: bronze/hevy/raw_workouts.parquet e bronze/users/raw_form.parquet.
  • Importante para auditoria, reprocessamento e trilhas de governança.
Silver – Trusted
  • Dados limpos, padronizados e tipados, com atenção a datas, unidades de medida e identificadores.
  • Tabelas de usuários, treinos, séries e exercícios integradas, com chaves consistentes para relacionamentos.
  • Aplicação de regras de qualidade (remoção de duplicidades, preenchimento de valores ausentes quando possível, validação de faixas plausíveis).
Gold – Curated
  • Star Schema com tabelão de métricas de treino: fact_workout_metrics + dimensões de usuário, data e exercício.
  • Estrutura pensada tanto para análise OLAP quanto para servir de base a uma possível Wide Table para machine learning, em linha com as recomendações da disciplina.
  • Camada que alimenta dashboards, notebooks analíticos e o app Streamlit.

A divisão em Bronze, Silver e Gold facilita o entendimento do fluxo de maturidade dos dados: de “apenas coletado” para “confiável” e, por fim, “pronto para gerar valor de negócio”.

Slide 6 · Camadas do Data Lake Próximo: Airflow orquestrando o ETL
Orquestração

Airflow como maestro do pipeline

Para atender ao requisito de integração e automação, o projeto utiliza o Apache Airflow como componente central de orquestração. Em vez de rodar scripts independentes, o fluxo de dados é descrito como uma DAG (Directed Acyclic Graph), que deixa explícita a ordem correta das atividades, as dependências entre elas e os pontos de falha possíveis.

DAG principal – instructor_workout_pipeline
  • create_lake_structure – cria a estrutura de pastas do Data Lake (Bronze, Silver, Gold) no S3.
  • ingest_users_form, ingest_hevy, ingest_kaggle – tarefas que fazem a ingestão das diferentes fontes para a camada Bronze, podendo rodar em paralelo.
  • silver_users, silver_transform, silver_kaggle – transformações em Python/Pandas que elevam a qualidade dos dados para a Silver.
  • gold_metrics – etapa final que gera o Star Schema e o tabelão de métricas consumido pelo app e por ferramentas de BI.

O Airflow registra logs detalhados, permite tentativas automáticas em caso de erro e facilita o reprocessamento de um intervalo de tempo específico, algo essencial em pipelines de dados em produção.

Aspectos práticos
  • Cada task utiliza BashOperator para acionar scripts Python versionados no GitHub.
  • O pipeline pode ser agendado (por exemplo, execuções diárias) ou acionado manualmente para reprocessamentos pontuais.
  • Se uma ingestão falha, as tasks dependentes não são executadas, evitando que a camada Gold seja alimentada com dados incompletos.
  • A interface web do Airflow exibe o grafo, o histórico de execuções e os tempos de cada etapa, contribuindo para observabilidade.
Slide 7 · Orquestração com Airflow Próximo: Modelagem Star Schema
Modelagem de Dados

Star Schema e preparação para Wide Table

Tabela Fato – fact_workout_metrics
  • Grão definido como métricas por usuário, data e exercício.
  • Métricas principais: total de séries, repetições, volume (kg), carga máxima, duração, número de treinos por período.
  • Chaves técnicas conectando às dimensões: user_key, date_key, exercise_key.
Dimensões
  • dim_user: informações de perfil, objetivos, nível de experiência, eventuais restrições médicas.
  • dim_date: data completa, semana, mês, ano, dia da semana e identificadores úteis para análises sazonais.
  • dim_exercise: nome do exercício, grupo muscular, tipo de equipamento, se é composto ou isolado, entre outros atributos.
Conexão com o Hands-on
  • O Star Schema atende ao requisito de análise OLAP destacado pelo professor.
  • A partir dessa estrutura, é possível construir uma Wide Table para alimentar modelos de machine learning focados em predição de aderência ou performance.
  • O mesmo modelo pode ser consumido por ferramentas de BI corporativas, sem necessidade de reformulação completa.

A modelagem reforça o storytelling de evolução: os dados saem de registros transacionais de treinos e são reorganizados em torno de perguntas analíticas sobre carga, consistência e progresso ao longo do tempo.

Slide 8 · Modelagem Star Schema Próximo: MVP com Streamlit & IA
Serving Layer

MVP: App Streamlit e Chat IA

A camada de consumo conecta a arquitetura técnica ao usuário final. Em vez de expor apenas tabelas e arquivos, o projeto entrega uma aplicação que fala a linguagem de quem treina: telas simples, métricas claras e um assistente virtual que entende o contexto de treino.

Streamlit App
  • Login com usuários armazenados em users_app.parquet no S3, permitindo simulação de múltiplos perfis.
  • Formulário que alimenta a pasta user_profiles/ com informações de objetivo, rotina e restrições.
  • Dashboard que lê a Gold e apresenta evolução de volume, carga máxima e frequência em nível de usuário.
Chat IA – Personal Trainer
  • Implementado com modelo LLaMA 3.1 via Groq API, consumido a partir do app.
  • O contexto da conversa inclui informações de perfil e recortes da fact_workout_metrics, permitindo respostas personalizadas.
  • Exemplos de uso: sugestões de ajustes semanais, divisão de treino, dicas de progressão de carga.
Dados de Teste e Demonstração
  • Script generate_fake_test_data.py gera fact_workouts_test_user.parquet com um usuário fictício.
  • Usuário de demonstração: testuser@example.com / 123456, já com histórico preenchido.
  • Esse arranjo permite demonstrar todas as funcionalidades, mesmo em cenários sem acesso a APIs externas em tempo real.
Experiência do usuário
  • O usuário deixa de ver seus treinos como registros isolados e passa a enxergar uma narrativa: onde estava, onde está e para onde pode ir.
  • O instrutor tem um ponto único de consulta para analisar padrões de treino e sugerir adaptações.
  • A IA entra como camada de suporte, traduzindo métricas em recomendações de treino em linguagem natural.
Slide 9 · MVP Streamlit & IA Próximo: Governança, Qualidade e Automação
Boas Práticas

Governança, Qualidade e Automação

Governança & LGPD
  • Estrutura de pastas e tabelas organizada por camada e domínio, ex.: bronze/hevy, silver/users, gold/fact_workout_metrics.
  • Uso mínimo de dados pessoais, priorizando identificadores técnicos e informações de treino; dados sensíveis podem ser ofuscados ou pseudonimizados.
  • Controles de acesso e criptografia em repouso podem ser implementados via políticas de IAM e configurações do S3 em um cenário de produção.
Qualidade de Dados
  • Validações em Silver para garantir tipos corretos, ranges plausíveis e coerência entre entidades (por exemplo, séries não negativas, datas dentro de períodos válidos).
  • Deduplicação de registros por chaves de negócio e timestamps para evitar contagens infladas de volume ou frequência.
  • Padronização de cálculos de métricas, garantindo que “volume de treino” signifique a mesma coisa em todos os relatórios.
CI/CD & IaC – Entregáveis do Hands-on
  • GitHub como repositório de versionamento do código de ETL, do Airflow e do app Streamlit, atendendo ao requisito de documentação e controle.
  • Proposta de pipeline CI/CD com GitHub Actions para rodar testes, lint e build de imagens Docker a cada alteração relevante.
  • Uso sugerido de Terraform para provisionar buckets S3, usuários IAM e infraestrutura de execução, alinhando o projeto ao conceito de Infraestrutura como Código descrito no material.

Mesmo em um projeto acadêmico, a preocupação com governança, qualidade e automação aproxima a solução dos desafios reais enfrentados por times de Engenharia de Dados em produção.

Resumo do storytelling técnico

O trabalho do Instructor Workout conecta diretamente os pilares do Hands-on de Engenharia de Dados em Big Data ao contexto de treinos físicos: partindo de um problema de negócio claro, passando pela construção de uma arquitetura Lambda em nuvem, pela criação de um Data Lake governado e pela modelagem analítica, até chegar a um MVP de aplicação com IA focado no usuário.

Pontos-chave
  • Arquitetura em camadas (Bronze, Silver, Gold) alinhada às boas práticas de Engenharia de Dados.
  • Orquestração com Airflow, permitindo automação, observabilidade e reprocessamentos seguros.
  • Modelagem Star Schema preparada para evoluir para cenários de machine learning (Wide Table).
Aprendizados
  • Importância de começar pelo problema de negócio e não pela tecnologia.
  • Benefícios de separar ingestão, transformação, armazenamento e consumo em componentes bem definidos.
  • Valor de um MVP funcional para demonstrar, na prática, o impacto de uma arquitetura de dados bem desenhada.
Próximas Explorações
  • Incluir novas fontes (por exemplo, dados de sono, nutrição ou wearables), ampliando a visão 360º do usuário.
  • Explorar personalização baseada em grupos de usuários semelhantes (clusterização) e trilhas de treino automatizadas.
  • Transformar o MVP em um produto completo, com monitoramento, alertas e integrações com academias e plataformas de fitness.
